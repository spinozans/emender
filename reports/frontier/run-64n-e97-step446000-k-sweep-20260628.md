## Recommendation

Do not mix these results with earlier step-383500 runs. This comparison is only for the fixed step-446000 checkpoint.

For the next 64-node refinement, keep the same checkpoint and run one of:

1. K80 hierarchical group_size=8, bucket 64M, avg outer optimizer. This is the preferred next test: K80 is the best terminal arm on both loss-improvement windows, and group_size=8 directly tests whether the current winner can reduce hierarchical overhead without losing learning speed.
2. A longer 60 minute K40/K80/K160 comparison only if the next decision requires separating K40's slightly lower terminal trainer loss from K80's better improvement-rate efficiency. K320 should not be included as a primary contender unless there is a specific need for a low-merge negative-control row.

Do not switch to schedule-free outer changes next. This sweep held the local optimizer fixed as schedulefree and used avg outer updates; the observed spread is primarily K/merge-cadence behavior, not evidence that the outer optimizer should change. Test K80 group_size=8 first, then decide whether schedule-free outer deserves a separate controlled probe.

Do not unpause a 256-node extended candidate yet. The completed 64-node evidence supports K80 as the best candidate to refine, but the next scale step should wait for one focused 64-node K80 topology refinement rather than carrying a known-weak K320 cadence upward.
